# 浮生万象：AI 开放式人生模拟器

React + TypeScript + Vite。流程为：配置 AI → 选择模板 → AI 生成严格 10 项主题 → 选择 3—5 项 → AI 生成完整世界 → 投胎与家庭 → 角色（姓名、性别）/天赋 → AI 出生叙事。

## AI 配置与隐私

应用**不含本地伪 AI、不含默认 endpoint、Key 或模型，也没有本地生成 fallback**。开始前须在“AI 设置”中自行填写：

- API 基础地址（通常填写到 `/v1`）：OpenAI 自动请求 `/chat/completions` 和 `/models`，Anthropic 自动请求 `/messages` 和 `/models`
- 即使误填完整的 `/chat/completions`、`/messages` 或 `/models`，也会先规范化，避免重复路径
- API Key 与模型；可点击“拉取模型”从服务端读取，也可继续手动填写
- 最大输出 token、temperature 均为可选输入；输入框留空时分别按 `8192` 和 `0.7` 发送，填写后使用自定义值

配置只存储在浏览器 `localStorage` 的独立键中，不进入 `SimulatorState`/游戏存档，也不会被源码提交。真实密钥仍会由浏览器直接发送到用户填写的 API 地址，因此应仅使用可信服务和设备，并确保服务支持浏览器 CORS。设置支持显示/隐藏 Key、拉取模型、测试连接、保存和清空。模型拉取成功后使用原生下拉选择框（手机可直接点选），并可随时切换“手动填写”输入任意模型；切换 API 格式、地址或清空配置会重置已拉取列表。模型接口兼容 `data: [{id}]`、`models: [{id/name}]` 以及字符串/对象数组，并会去重排序。Anthropic 服务若强制要求 `max_tokens`，在该项留空时将由 API 返回明确错误。

所有主题、结构化世界和出生篇章都由统一的 `RemoteAiProvider` 调用真实 API。主题标签由 AI 每次即兴生成，并加入独立创作扰动，不存在内置标签表；标签必须恰好 10 个且互不重复。其中约 5—6 项组成两组可连续发展的主线/矛盾链，其余约 4—5 项保持独立以形成支线与意外，标签本身不显示分组。

现代都市、古代王朝、武侠江湖、修仙世界、末日生存、赛博朋克、魔法学院、太空殖民等内容只是世界模板。模板只限定时代背景、力量体系和基本逻辑边界，不提供固定剧情。世界推演只会向 AI 发送模板边界、氛围规则、自定义补充和玩家选中的 3—5 个主题，不会把固定家庭、天赋或事件当作答案提示。结构化响应会去除 Markdown 代码块并从附加文本中提取 JSON；世界字段不完整时，UI 会显示错误并允许返回设置。

## 代码结构

- `src/ai/provider.ts`：OpenAI/Anthropic 请求、JSON 提取、校验及统一 provider
- `src/storage/aiSettings.ts`：独立的 AI 配置 localStorage
- `src/types.ts`：配置、世界、provider 与流程类型
- `src/App.tsx`：流程和 AI 设置弹窗
- `src/domain/generators.ts`：家庭、天赋及选择限制（不承担 AI 内容生成）

## AI 开局生成

投胎仅支持随机投胎、三选一和 AI 问答。家庭与天赋不再读取模板中的固定候选池：世界生成的一次请求会同时返回完整世界、严格 12 个多样且 ID 稳定的家庭候选池，以及 4 道世界专属出生问答（每题严格 4 个选项）。新世界缺少这些字段会校验失败，不使用固定问题或本地内容 fallback。三选一、刷新、随机投胎和问答匹配均在该池中本地完成，不再额外调用 AI。家庭候选必须是结构完整且同批 `label + parents + location` 组合互不重复的对象；天赋响应必须是恰好 10 个互不重复的字符串。随机投胎使用 Web Crypto 无偏随机索引锁定一个家庭；三选一每次抽取三个不重复候选；问答按生成选项与家庭文本相关度排序三个候选。家庭与天赋均可刷新，刷新会清空原选择并记录次数；天赋严格 10 选 3。

旧本地存档会自动补齐问答答案和刷新计数；已经废弃的 `custom` 投胎方式会被忽略。

## 人生数据底座（阶段二第一批）

出生后会创建结构化 `LifeState`：月龄/人生阶段、存活状态、8 项基础属性、健康/精力/心情/压力、个人与家庭资产、地点、家庭成员及时间线。`src/life/engine.ts` 提供可测试的纯函数 `createInitialLifeState`、`advanceLife`、`getLifeStage`、`formatAge`；推进支持 1/3/6 个月，普通月份不会刷时间线，仅阶段变化与周岁留下记录。旧的出生结果或人生首页存档会自动补建该状态，重新选择世界、家庭、角色或天赋会使旧人生状态失效。

## AI 事件驱动人生（阶段二）
- 单一动态推进按钮会先计算年龄、阶段与人物年龄；AI 事件成功生成后才原子写入存档，失败不会推进。
- 普通人生事件以一次 AI 请求严格批量生成 10 个通用蓝图并存入 `LifeState.eventQueue`；每次推进消耗一个，并按推进后的实时状态重新校验、过滤失效成员 ID、替换动态月数。缓存为空才再批量请求。待突破时不消费缓存，而按实时状态单独生成突破事件。
- 每次推进必须解决一个含 2—4 个选择的结构化事件。普通选择直接使用事件、选择与本地效果摘要，不再逐次请求 AI 润色；只有突破结果保留可选 AI 叙事。所有效果由本地白名单校验、限幅和应用，AI 无权写任意状态路径。
- 支持家庭与社交人物、关系、NPC/玩家死亡、按世界寿命推断的自然上限和自然死亡；时间线默认仅展示最近 20 条。

### 动态时间推进
人生首页只提供一个动态推进按钮。新生儿、幼儿与童年保持细粒度；普通寿命世界成年后按年推进，衰老或健康恶化时自动缩短；数百年至数千年的长寿/修仙世界成年后可按 5、10、25 或 50 年推进，并在接近寿命上限时减速。寿命尺度从生成世界的 `lifespan` 文本推断并存入存档；旧存档会自动补齐。当前人生阶段划分仍按固定年龄，长寿世界可能很早显示“成年”，但动态跨度会依据其寿命尺度生效。

## 世界成长体系与寿命

每个新生成世界必须包含严格的 `progressionSystem`：3—12 层、首层进度为 0、后续门槛严格递增，并声明 `increasing` / `decreasing` / `mixed` / `unchanged` 寿命模式。每层分别包含寿命年数、不老与自然死亡开关、突破条件和风险；世界确认页可只重新生成成长体系而不改动其他世界字段。正常的新世界生成若缺少任何字段会直接报错，本地 fallback 仅作为测试和内部保护，不参与成功的 AI 世界生成。

角色寿命始终来自当前层。事件的 `modify_progression` 达到下一层门槛后只会进入“待突破”状态，不会自动晋升；下一次短跨度推进必须生成突破事件。每个突破选项都由本地依据意志、幸运、体魄、健康、风险策略和随机数裁决，AI 不能直接修改层级。成功后才更新层级与寿命；失败会损失进度、健康和精力，冒险策略还可能致死。寿命降低到低于当前年龄时会因晋升代价耗尽寿元；不老层关闭自然年龄死亡，但健康归零和事件死亡仍然有效。突破事件结果先在本地完成真实裁决，再可把实际结果交给 AI 润色；普通事件不请求结果叙事。时间线效果摘要将属性、生命状态、账户和成长标签统一显示为中文。
